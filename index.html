Легко ли вам оценивать?

Время = деньги

Почему еще важно

  • Продать клиенту свое время
  • Объем работ должен быть конечен
  • Планирование загрузки разработчиков

Виды оценки

  • Предварительная оценка — высокоуровневая оценка
  • Точная оценка — оценка конкретной задачи
  • Переоценка — изменились требования, поменялся разработчик

Причины ошибок в оценках

  • Неточные требования или неизвестный набор задач
  • Не учитываются риски
  • Сложно оценить то, чего никогда не делал
  • Люди ошибаются в своих оценках, так как они - люди, а не оракулы
  • Недостаточное понимание ответственности
  • Производительность разработчиков может отличаться разы

Как научиться

0. Трекать время

  • The 20 Best Time-Tracking Apps (статья)

1. Уточняйте требования

  • Не бойтесь задавать вопросы
  • Убедитесь, что все моменты учтены
  • Давайте интервал вместо точной цифры

2. Детализируйте большие задачи

  • Разбивайте задачу на очевидные подзадачи
  • Думайте над (конкретной) реализацией
  • Не увлекайтесь дроблением

3. Оценивайте командой

  • Давайте задачи оценку нескольким специалистам
  • Дайте на ревью более опытному разработчику
  • Берите усредненные цифры для неточных оценок

4. Делайте переоценку задач

  • Подтвержайте заложенную оценку перед началом работы
  • Распишите/обсудите конкретную реализацию задачи

5. Определяйте рисковые задачи

  • Особо неопределенные задачи — новая технология, внешнее API и т.д.
  • Отдельно выделяйте время на research таких задач
  • Проговаривайте риски с менеджером/клиентом
  • Затем, если возможно, оценивайте/переоценивайте

6. Не забывайте про буфер

  • Помните, что вы никогда не тратите все 100% рабочего времени на задачу
    • Интеграции
    • Деплой, git
    • Коммуникации
    • Тестирование
    • Кофе, туалет, проверка почты, общение с коллегами, прокрастинация

7. Проблемы? Не молчите!

  • Советуйтесь с командой и не бойтесь просить помощи
  • Клиент — тоже человек, его можно переубедить

8. Собирайте статистику

  • Оценка - это не только то, что будет
  • Анализируйте оцененное и реально затраченное время
  • Проводите "ретро" по переоценным или недооцененным задачам
  • Накапливайте опыт для последующих оценок
  • Учитывайте коэффициент погрешности оценок (реальное / оцененное)

Бонус. Повышение прогнозируемости

  • Улучшайте комуникацию в команде
  • Как только появилась задержки по срокам - сразу к менеджеру
  • Комбинируйте людей в команды так, чтобы им было комфортно
  • Группируйте людей (по отделам, по проектам)
  • Пробуйте различные техники программирования (парное, экстремальное)
